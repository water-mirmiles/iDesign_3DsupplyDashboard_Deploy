# Supply3D（iDesign 3D Supply Dashboard）产品需求文档 PRD — V1.0 验收版

**文档状态**：与当前主干实现一致，作为 V1.0 正式验收与交付说明。  
**产品代号**：Supply3D  
**核心目标**：在鞋履供应链场景下，追踪**款号 / SKC** 与 **3D 楦（Last）、3D 大底（Sole）** 资产的匹配与覆盖进度，支撑数据驱动的补资产与排产决策。

---

## 1. 产品概述

### 1.1 业务价值
- 统一查看多数据源（Excel 宽表）与物理 3D 资产目录的对账结果。
- 将「是否有楦号/底号」「是否已有 3D 文件」「覆盖多少生效款」量化为主屏 KPI 与可下钻清单。
- 通过受控账号体系，保证映射变更、同步与上传行为**可审计**。

### 1.2 用户角色
| 角色 | 能力概要 |
|------|----------|
| **超级管理员 (`admin.water`)** | 用户体系管理：创建用户、激活/管控、重置密码；绕过部分运维限制。 |
| **业务用户** | 登录后查看 Dashboard、清单、3D 预览；在授权范围内触发同步/上传（以当前后端策略为准）。 |

### 1.3 核心业务规则（与实现一致）
- **三级关联**：款号 (Style) → SKC（款+色+材等业务键）→ 资产（楦码 `lastCode` / 底码 `soleCode`）。
- **3D 匹配**：以**归一化后的文件名/编号**与清单中的楦码、底码对账；支持横杠/空白/不可见字符归一及扩展名大小写不敏感。
- **统计口径**：Dashboard 关键指标对「生效」款号的处理与 `data_status === 'active'`（及上游 Excel 状态清洗结果）对齐；支持按款式状态、产品定级等多维筛选。
- **时间维度**：支持基于数据快照日期的趋势与对比（具体粒度以实现为准）。

## 2. 技术架构（V1.0）

### 2.1 总体结构
- **单体部署形态**：**Node.js + Express** 同时提供 **REST API** 与 **前端静态资源**（Vite 构建产物 `dist/`），默认 **`http://<host>:3001`** 单端口访问（开发期前端可另起 dev server，生产以「单端口」为验收标准）。
- **存储**：以 **文件系统** 为权威存储介质，辅以轻量级 **JSON 缓存**（无独立数据库进程要求）。

### 2.2 后端职责
- 托管 `dist/`（HTML/JS/CSS）。
- 挂载 **`/storage`** 静态目录，指向运行时解析的 **`server/storage` 绝对路径**（与 `process.cwd()` 对齐），对外提供 3D 与上传文件直链。
- 提供上传、认证、用户管理、看板聚合、清单、资产元数据与详情等 API。

### 2.3 前端职责
- React + Tailwind + Lucide + Recharts；**全宽布局**（`w-full`），避免无故限宽。
- 生产环境 **API 与静态资源请求与当前页面同源**（相对路径），避免硬编码 `localhost` 导致跨环境 404。

### 2.4 关键持久化文件（示例）
| 路径（相对 `server/storage`） | 用途 |
|-------------------------------|------|
| `auth/users.json` | 受控用户与密码哈希（及角色信息，以实现为准） |
| `mapping_config.json` | 映射与 Join 配置落盘，支撑无 UI 配置冷启动 |
| `final_dashboard_data.json` | 看板/清单主快照 |
| `full_inventory_cache.json` | 全量清单与增量对账缓存（若启用） |
| `assets_meta.json` | 3D 预处理测绘元数据索引 |
| `upload_audit.json` | 上传审计（路径 → 操作者/时间） |
| `modeling_debug.log` / `ai_trace.log` 等 | 排障与 AI 追踪日志 |

## 3. 功能模块

### 3.1 身份与权限系统
**目标**：企业内网可用的**管理员受控**账号体系，非开放自助注册（策略以实现为准）。

**能力清单**
- **登录**：用户名 + 密码；服务端 **SHA-256** 校验；前端生产环境不向错误主机发送请求。
- **超级管理员 `admin.water`**：种子账号；**仅该账号**可管理用户列表（创建、状态、重置密码等），详见后端路由与中间件。
- **审计字段**：数据重算、同步等操作可携带 **`updatedBy`（当前用户）** 与精确时间，便于清单与日志追溯。

**验收要点**
- 非管理员无法调用用户管理 API。
- 用户数据存放在受保护路径，**缓存清理逻辑不得删除 `auth/`**（已实现白名单清理）。

---

### 3.2 映射引擎（AI-first）

**目标**：在 **17 张 SQL DDL** 约束下，完成从「原始 Excel 列」到「标准业务字段」的映射，并自动推导多表 **Join 路径**，降低手工配置成本。

**能力清单**
- **DDL 驱动解析**：以表结构 + **COMMENT** 为语义优先来源，生成可用字段字典与映射建议。
- **Golden Record / Chat-to-Config**
  - 通过对话或样例描述，引导生成/修订映射与 Join 假设。
  - 结合 **黄金样本 XLSX** 对 Join 路径做校验与纠偏（以实现函数为准，如 `validateJoinPathSuggestionsWithGoldenXlsx` 等）。
- **逻辑持久化**
  - 映射与关键引擎参数写入 **`mapping_config.json`**。
  - 服务重启后无需重新在 UI 点选即可按上次认证配置聚合（在无新上传/强制刷新前提下）。

**验收要点**
- 映射变更可保存、可复现；主表切换与 Join 路径错误时有明确日志或 UI 提示。
- Vertex / AI 调用失败时，系统**降级行为可预期**（例如保留上次配置、提示重试），不因单次 429 导致不可恢复状态。

---

### 3.3 3D 资产处理流水线

**目标**：上传与扫描 **Last / Sole** 目录下的 **OBJ / STL / GLB / 3DM**（以实现支持的扩展名为准），完成**可预览、可测绘、可统计**的闭环。

**子能力**

1. **预处理**
   - **OBJ → GLB**：服务端 `obj2gltf` 异步管线；失败时写入 **`modeling_debug.log`**，并允许仅 OBJ 预览。
   - 上传后 **权限归一**（如 `0644`），避免进程/用户组不可读。

2. **自动测绘**
   - 对 GLB（及管线产出物）提取 **L/W/H** 等工业口径尺寸、体积/建议受众码段等（与 `last3dMetrics` 模块一致）。
   - 元数据写入 **`assets_meta.json`**，供前端跳过重复几何扫描。

3. **物理发现与对账**
   - 目录扫描使用统一 **`normalizeAssetCodeKey`**；忽略 **`._*`** Mac 影子文件干扰。
   - 增量同步：扫描 `storage/assets/` 更新清单中的 `has3D` / 匹配状态（以实现 API 为准）。

4. **双轨加载（前端）**
   - **优先 GLB**（带超时/HEAD 校验）；失败或不可达时 **自动降级 OBJ/STL**。
   - 资源 URL **带缓存穿透参数**（如 `t=`），减少错误 CDN/浏览器缓存干扰。
   - 生产环境静态请求与 **当前站点同源**，避免 localhost 分裂。

**验收要点**
- 同编号在列表、弹窗、物理目录三者一致；上传后无需手工改路径即可预览。
- `GET /api/asset-details` 返回的 `glbUrl` / `objUrl` / `file.previewUrl` 与 `/storage` 实际可访问一致。

---

### 3.4 数据仪表盘（Dashboard）

**目标**：一屏展示规模、覆盖与结构，并支持管理层关心的筛选口径。

**能力清单**
- **多维筛选**
  - **款式/数据状态**（生效/草稿/作废等，以清洗后枚举为准）。
  - **产品定级**（`product_actual_position` 等；S/A/B/C 与「其他」分层交互，以实现为准）。
- **KPI 与趋势**
  - 区分 **唯一 3D 资产数** 与 **覆盖款号数** 等口径（避免重复计数误导）。
  - Recharts 展示品牌、趋势等（数据来自后端聚合快照，非前端 Mock）。
- **三层嵌套进度叙事**
  - **Total → HasCode → Has3D**：从「总款/生效池」→「已关联业务编码」→「已命中物理 3D」的漏斗或分层指标（以实现组件文案与图表为准）。

**验收要点**
- 筛选变更后 KPI 与图表联动刷新，且无「假延迟加载」（禁止纯 `setTimeout` 模拟异步）。

---

### 3.5 清单管理（Inventory）

**目标**：高性能浏览与筛选上万行级清单，并支持 3D 快速核验。

**能力清单**
- **搜索与历史**：本地持久化 **最近 10 条** 搜索历史；支持聚焦/下拉展示与单条删除、清空（以实现为准）。
- **高级筛选**：侧栏/抽屉内多选：定级、品牌、资产状态、数据状态、日期区间等；**AND** 逻辑组合。
- **表格结构**：产品定级独立列、排序；楦/底 3D 状态列（已匹配/缺失）可点击。
- **预览弹窗（约 7:3 布局）**
  - 左侧约 **70%**：Three.js 画布（GLB 优先 + OBJ/STL 降级）。
  - 右侧约 **30%**：文件信息、上传者、**关联款号列表**、楦-底关联列表、下载与「重新预处理」等。
  - 关联列表与列表页条数一致：当物理文件存在而缓存为 0 时，后端 **强制现场全量重算** 再返回。

**验收要点**
- 弹窗请求与列表 **同源**，无 `localhost` 硬编码分裂。
- 预览与下载 URL 与服务器 `/storage` 一致。

## 4. 非功能需求（V1.0）

| 类别 | 要求 |
|------|------|
| **可部署性** | Linux 服务器单进程可启动；`cwd` 与 `server/storage` 关系明确；启动日志可核对静态根目录。 |
| **可观测性** | 关键链路日志：`[Static-Check]`、`[Discovery]`、`[Logic-Audit]`、`[Realtime-Link]`、上传追踪、建模失败日志。 |
| **安全** | 密码哈希存储；管理员操作隔离；避免在 JSON 缓存中写入不可移植的绝对路径。 |
| **性能** | 全量聚合成本高时使用快照与增量 3D 同步；避免无意义重复全量（策略以实现为准）。 |

## 5. 验收检查清单（摘要）

1. **登录与用户管理**：`admin.water` 可管用户；普通用户不可越权。
2. **映射闭环**：DDL + Golden 样例 + `mapping_config.json` 可驱动一次完整 `processAllData`。
3. **3D 闭环**：上传 OBJ →（可选）GLB → 预览 → 测绘元数据 → Dashboard/清单状态更新。
4. **Dashboard**：多维筛选 + 嵌套进度叙事 + 口径与 `data_status` 一致。
5. **清单**：搜索历史、高级筛选、7:3 预览、关联款数与列表一致。
6. **部署**：单端口 3001 可访问；`/storage` 与 API 同源无 404。
